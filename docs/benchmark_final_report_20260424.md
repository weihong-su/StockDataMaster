### 3.2 双源校验行级通过率分布

基于 404 只参与双源校验的股票（冷+热启动合并）：

| 通过率区间 | 股票数 | 占比 | 说明 |

**平均行级通过率：48.9%**

### 3.3 根因：前复权系数差异

最常见价格差异模式揭示了系统性问题：

```
价格差异分布:
  0-0.1 元:   11,945 行 (11.7%)  ← 浮点精度
  0.1-0.2 元: 18,273 行 (18.0%)  ← 单次除权差异
  0.2-0.5 元: 30,928 行 (30.4%)  ← 多次除权累积
  0.5-1.0 元: 19,695 行 (19.4%)  ← 大额除权
  1.0-5.0 元: 15,832 行 (15.6%)  ← 多次大额除权
  >5.0 元:     5,049 行  (5.0%)  ← 极端案例（高价股长历史）

最常见差值（均为整数 ×0.05 倍数）：
  0.20 元: 4,090 次
  0.10 元: 3,909 次
  0.50 元: 3,244 次
  0.40 元: 2,456 次
```

**结论：** tushare 和 xtquant 使用不同的前复权基准日（通常差 0.1～0.5 元/股），这是行业已知的不可避免差异。验证层的严格要求（行级精确匹配）过于苛刻，导致本来数据质量正常的股票被误判为校验失败。

### 3.4 热启动校验异常

```
热启动校验通过: 2 只（异常低）
热启动校验失败: 333 只
热启动降级直缓: 85 只
热启动增量缓存激活: 402 只
```

**原因链条：**
1. 热启动的每只股票有 4-45 条"增量新数据"（tushare 返回了比缓存更多的新日期）
2. 增量数据需要校验，请求 xtquant 获取对应日期段数据
3. xtquant 本地数据库仅到 2025-07（平均），无法提供 2025-08 至 2026-04 的数据
4. baostock 全程网络故障，无法备用
5. 结果：增量数据无法被双源校验 → 失败 → 不进入缓存

## 4. 增量缓存机制验证

### 4.1 功能正确性

```
冷启动后缓存: 96,867 条已验证，420 只股票
热启动激活增量缓存: 402 只股票
热启动后缓存: 97,201 条已验证（净增 334 条）
```

增量缓存机制 **功能正确**：
- 402 只股票的历史数据（占 96,867 条）无需重复校验
- 只对新增的几十条日期运行校验流程
- 增量范围精确（`incremental_start` ~ `incremental_end`）

### 4.2 缓存命中率 = 0% 的根因

benchmark 统计的缓存命中率为 0% 并非增量缓存失效，而是 **benchmark 脚本的 date_range 设计导致**：

```python
# benchmark_cache_validation.py:390-391
end_date   = today                     # 2026-04-24
start_date = today - 750 days          # 2024-01-29
```

tushare 返回 500 条，覆盖 2024-04-08 ~ 2026-04-24。  
但请求 start_date = 2024-01-29，`_is_date_range_covered` 判断：  
`cached_start (2024-04-08) > request_start (2024-01-29)` → **False** → 跳过缓存。

这是 benchmark 测试设计的问题，不是增量缓存的 bug。实际使用中如果请求 `count=500` 而非 `start_date`，热启动会直接命中缓存（sub-100ms 响应）。

### 4.3 真实热启动性能（已验证）

前一次小规模测试（2026-04-24 17:13）数据：

| 操作 | 响应时间 |
|------|----------|
| 首次10条（冷启动） | ~6,000 ms |
| 再次10条（热启动） | **<100 ms** |
| 扩展50条（增量校验45条） | ~5,000 ms |

**加速比：60x**（热启动 vs 冷启动）

## 5. 关键洞察与问题优先级

### P0 - 必须修复（影响热启动校验覆盖率）

**[问题] xtquant 本地数据库严重过旧（平均 270 天）**

现象：331 只股票 xtquant 数据在 2025 年某时点停止更新，导致热启动增量数据（2025 后半年至 2026）无法被校验。

根因：QMT 客户端未配置自动数据同步，或 QMT 账户权限限制。

解决方向：
```python
# 选项 A: 强制刷新模式 — 在 download_history_data 时不使用 incrementally=True
# 选项 B: 在校验前检测数据时效性，过旧则延长下载等待或跳过此校验
# 选项 C: xtquant 降级为"仅当数据新鲜时作为校验源"，否则只用 baostock
```

### P1 - 重要优化（消除系统性校验假阳性）

**[问题] 前复权校验标准过严（tushare vs xtquant qfq 基准日不同）**

现象：71.3% 的参与校验股票有 1-50% 行级不通过，主因是两者前复权系数基准不同，导致历史价格相差固定的 0.1-0.5 元/股。

解决方向：
```python
# 方向 A: 改用相对价格比较（(price_a - price_b) / price_b < 0.5%），替代绝对差值
# 方向 B: 仅校验 close 价格（市场共识），不校验 open（时效性敏感）
# 方向 C: 允许"持续一致差异"（每行差值标准差 < 0.1 时视为通过）
```

**预期效果：** 校验通过率可从 17.1% (100% 一致) 提升至 70%+ （引入 ±1% 相对容差后）。

### P2 - 稳健性优化

**[问题] baostock 长期网络不可用**

本次测试整轮 902 次全失败，排除一个校验源。

解决方向：
- 在每次校验前先做 10 秒 baostock 健康探针，不可用时立即跳过（当前机制每次都尝试并等待超时）
- 将 baostock 超时缩短至 3 秒

**[问题] benchmark 脚本 start_date 过早，导致缓存被绕过**

修复方式：
```python
# 改为使用 count 参数而非 start_date/end_date
df = dm.get_kline(code, freq='d', count=500, adjust='qfq', use_cache=True)
```
